# Tauri shell

Native desktop wrapper. Tauri v2, Rust backend. **The desktop shell does not run games.** Manabrew games run in WASM (the same web worker the browser build uses), and lobby/relay traffic runs through the JS WebSocket client. Tauri's only game-specific job is hosting a **Forge** engine room via the bundled self-hosted-node.

Read first: `/AGENTS.md`.

## Layout

| File                        | Role                                                                                                                                                  |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `src/lib.rs`, `src/main.rs` | Tauri entry — registers handlers, sets bundled-resource env vars, manages state.                                                                      |
| `src/asset_server.rs`       | `main_window_url` + `start_asset_server`: serve embedded assets over `http://localhost:9527` with COOP/COEP (macOS/Linux) for cross-origin isolation. |
| `src/commands.rs`           | `is_card_supported` — the one card-db command the UI still calls natively.                                                                            |
| `src/forge_room.rs`         | `start_forge_host` / `stop_forge_host` — spin up / tear down the bundled self-hosted-node Forge room (feature `forge-room`).                          |
| `src/local_relay.rs`        | `start_local_relay` / `stop_local_relay` — embedded `manabrew-server` relay on `127.0.0.1:<ephemeral>` for Forge play-vs-AI (feature `forge-room`).   |
| `src/card_db.rs`            | Card database loader (rkyv archive) backing `is_card_supported` and the limited stack.                                                                |
| `src/limited_*.rs`          | Sealed / draft / winston / gauntlet / cube — the limited-format command surface and its session manager.                                              |
| `tauri.conf.json`           | Bundle config (forge runtime + cardset archive) and the COOP/COEP headers that make WASM/SharedArrayBuffer work in the webview.                       |

## Conventions

- **The shell hosts the web frontend; the frontend runs the engine.** `src/platform/tauri.ts` (`TauriPlatform`) wraps a `WebPlatform`: `game`, `storage`, `events`, and `server` all delegate to it. The only override is `server.createRoom` with `engine === "Forge"` (and `stopRoom`), which call the native `start_forge_host` / `stop_forge_host` commands. `TauriPlatform.invoke` stays on native IPC for `is_card_supported` and the `limited_*` commands (the WASM worker only implements a subset of the limited surface).
- **Cross-origin isolation is mandatory → macOS/Linux packaged builds serve over a local http origin.** WASM needs `SharedArrayBuffer` (the worker blocks on `Atomics.wait` for human input — `game-engine.worker.ts`), which WebKit/WebKitGTK only allow when `crossOriginIsolated` is true (COOP `same-origin` + COEP `require-corp` **on an http(s) origin**). The packaged `tauri://localhost` custom scheme can't be cross-origin isolated and refuses `fetch`/Cache API (`Request url is not HTTP/HTTPS`) — no header fixes a custom scheme. So `asset_server.rs::start_asset_server` serves the embedded assets (`app.asset_resolver()`) over `http://localhost:<port>` with COOP/COEP and `asset_server::main_window_url` points the window there. This is the production equivalent of `tauri dev` (vite on `http://localhost:1420`); wasm SAB works because it's a real http origin. But to Tauri that origin is **remote**, which means the app's own commands (`start_forge_host`, `is_card_supported`, every `limited_*`) are ACL-gated — from a remote origin Tauri requires an explicit permission or the invoke is rejected with `"<cmd> not allowed. Plugin not found"`. The `remote` block in `capabilities/default.json` only **scopes** which origin a permission applies to; it grants nothing on its own. The actual grant is the `allow-app-commands` permission (`permissions/app-commands.toml`), referenced (unprefixed) in the capability. Because that capability is `local: true` **and** has the `remote` block, the one grant covers dev/Windows (local origin) and macOS/Linux (remote origin) at once. **Windows** keeps the default `http://tauri.localhost` embedded protocol (already http) with headers from `tauri.conf.json` → `app.security.headers`; **dev** gets them from `vite.config.ts` (`require-corp` under Tauri via `TAURI_ENV_PLATFORM`). `require-corp` means cross-origin subresources must be CORS-loaded — Scryfall images already set `crossOrigin="anonymous"`. Without isolation the engine can't start (`PlatformRuntimeChecks` in `App.tsx`).
- **Forge rooms need a live relay connection.** `start_forge_host` takes the relay `host` / `port` / `relay_password` as command args (passed from the JS relay client's current connection), then the local player joins the room it created through the normal web relay path. For play-vs-AI that relay can be the embedded loopback one: `start_local_relay` runs `manabrew_server::server::serve` in-process on `127.0.0.1:<ephemeral>` with a random per-run key and returns `{host, port, password}` for the JS client to connect to, so no external relay is required (multiplayer still uses one).
- **Bundled resources.** `scripts/harness.mjs` stages Forge engine assets into generated `src-tauri/resources/forge-runtime/`, with `cardsfolder.zip` matching Forge's packaged-resource lookup. Tauri copies that bundle to `engine/forge/`. `CARDSET_ARCHIVE` is the separate `cardset.rkyv` resource produced from `src-tauri/resources/cardset.rkyv` by `src-tauri/build.rs`. Runtime paths are exposed via env vars (`CARDS_DIR`, `TOKEN_SCRIPTS_DIR`, `EDITIONS_DIR`, `MANA_BREW_FORGE_ASSETS_DIR`, `CARDSET_ARCHIVE`) set in `lib.rs::run()` before any command runs. Don't read paths relative to the source tree at runtime — packaged builds don't have it.
- **Resources are split per-platform, and Android can't use them.** `bundle.resources` lives in the platform confs, **not** base `tauri.conf.json`: `forge-runtime` (64 MB, only the desktop Forge-room host reads it via `MANA_BREW_FORGE_ASSETS_DIR`) ships **desktop-only** (`tauri.{macos,windows,linux}.conf.json`); `cardset.rkyv` ships desktop + iOS (`tauri.ios.conf.json`). Mobile hosts no Forge rooms, so keeping `forge-runtime` out of the base keeps it out of the IPA/APK. On **iOS** `resource_dir()` is `<app>/assets`, so the env-map paths resolve and `cardset.rkyv` mmaps fine. On **Android** `resource_dir()` is the `asset://` APK URI — `File::open`/mmap can't read it, so `card_db.rs` embeds `cardset.rkyv` via `include_bytes!` (`cardset_archive_bytes()`, the one archive-bytes source shared with `limited_bootstrap.rs`) and Android gets no `tauri.android.conf.json` resources. Mobile also skips the GraalVM harness build: `tauri.{ios,android}.conf.json` override `beforeBuildCommand` to `yarn vite:build` (`tauri.android.conf.json` carries **only** that override — no `bundle.resources`, so Android bundles nothing and relies on the embedded archive).
- **Native Forge lib bundling (feature `forge-room`).** The Graal `native-image` library (`forge-harness/native/build/`, built by `build-native.{sh,ps1}`) is bundled per-platform via Tauri's auto-merged `tauri.{windows,macos}.conf.json` (the base `tauri.conf.json` has no native lib, so a Linux/no-feature build stays clean). Windows: `forgeharness.dll` lands next to the exe (no rpath; the loader searches the exe dir) — it is **load-time linked**, so a misplaced dll means the app won't launch. macOS: `libforgeharness.dylib` lands in `Contents/Resources`, ad-hoc signed in `build-native.sh` (arm64 won't load unsigned dylibs). The `@executable_path/../Resources` rpath that lets the `.app` resolve it **must** be emitted by `src-tauri/build.rs` (the executable crate) — a `cargo:rustc-link-arg` rpath from the `self-hosted-node` dependency does **not** propagate to the final binary, so relying on it ships an `.app` whose only rpath is the absolute build-machine path and crashes at launch with `Library not loaded: @rpath/libforgeharness.dylib`. Keep the two platform conf files and the `build.rs` per-OS link arms in sync.
- **Auto-updates.** `tauri-plugin-updater` + `tauri-plugin-process`, checked from the UI (`src/hooks/useDesktopUpdater.ts`, mounted in `AppShell`). Endpoint is `play.manabrew.app/tauri.json` (Caddy redirect to the latest GitHub release's `tauri.json`, generated by `scripts/gen-updater-manifest.mjs`); the `pubkey` in `tauri.conf.json` pairs with the `TAURI_SIGNING_PRIVATE_KEY` repo secret CI signs with — regenerating the keypair orphans every shipped install. `updater.conf.json` is a CI-only `--config` overlay enabling `createUpdaterArtifacts` so local `yarn tauri build` doesn't require the key. Updater trust is this minisign signature, independent of OS code signing — unsigned/ad-hoc builds still self-update because in-app downloads carry no quarantine/Mark-of-the-Web. The `updater:default` / `process:allow-restart` permissions in `capabilities/default.json` cover the remote asset-server origin like every other grant there.
- **Tauri command surface.** Every UI-facing command is registered in `lib.rs::run()` via `tauri::generate_handler![…]`. Adding a command means: implement, add to the handler list, expose from the appropriate module.
- **Long-running work runs off the command thread.** Tauri commands are async; spawn into the runtime, don't block.
