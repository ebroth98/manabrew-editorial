# Game UI — board, modals, panels, zones

The largest UI subtree. Read first: `src/AGENTS.md`, `docs/STYLE_GUIDELINES.md`, `docs/agents/UI_THEME_RULES.md`.

## Layout

| Folder / file                             | Role                                                                                                                                                                                                        |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `GameBoard.tsx`, `Game.tsx` (in `views/`) | Top-level board composition.                                                                                                                                                                                |
| `panels/`                                 | Player panels, mana pool, action log, prompt actions. The spell stack is rendered by Pixi (`@/pixi/stack/StackLayer`), not React — only the `SpellStackModal` ("View") + `StackSection` summary stay React. |
| `zones/`                                  | Small React overlays — `LibraryZoneTile`, `ManaAbilityTapButton`, `HandCardActions`. The battlefield and hand themselves are rendered by Pixi (`@/pixi/BoardCanvas`), not React.                            |
| `modals/`                                 | Prompt modals (target picker, choose mode, choose number, library peek, …). All built on the `Modal` compound. `PromptModalController` routes engine prompts to the right modal.                            |
| `cost-modals/`                            | Alternative-cost prompts (kicker, buyback, replicate, phyrexian, …).                                                                                                                                        |
| `dice/`                                   | Dice-roll UI: animations, feedback, and the reroll/swap/ignore/modify modals.                                                                                                                               |
| `game.types.ts`                           | Shared TS interfaces for board UI.                                                                                                                                                                          |
| `game.constants.ts`                       | Magic numbers, phase definitions, sizing values, color arrays.                                                                                                                                              |
| `game.styles.ts`                          | Reusable Tailwind class-string constants.                                                                                                                                                                   |
| `game.utils.ts`                           | Pure utilities (no React).                                                                                                                                                                                  |

When adding a constant, type, util, or class string, **check the shared modules first.** Don't duplicate.

## Board canvas — single unified Pixi scene

The whole table renders on **one** Pixi canvas, not one canvas per player. `BoardCanvas` (`@/pixi/BoardCanvas`) drives a single `BoardScene` (`@/pixi/board/BoardScene`) that hosts one `BoardRegion` per player, positioned by `computeBoardLayout` (`@/pixi/board/boardLayout`). Because every player's battlefield shares one coordinate space, cards can animate across the center line (MTGA-style combat staging) without crossing a canvas boundary.

- **Regions** (`BoardRegion`) own per-player grid layout, attachment stacking, rings, combat staging, and animation. The local region gets full interaction (drag/marquee/overlay); opponent regions are tap-to-target + hover only. Each reaches orchestrator services through a `RegionHost`.
- **Local controllers** live on the scene: `HandController`, `SelectionController`, `BattlefieldOverlay`, plus the `DragHandler` gesture. They talk to the scene through the `HandHost` / `SelectionHost` / `OverlayHost` seams in `@/pixi/board/types`.
- **Arrows + the spell stack** are not drawn inside `BoardScene`. A single transparent overlay canvas, `BoardOverlayCanvas` (`@/pixi/BoardOverlayCanvas`), sits _above_ the React panels (own Pixi app) and hosts both the `ArrowLayer` (pulls `BoardScene.getArrowDefs()` each tick) and the `StackLayer` (`@/pixi/stack/`). The canvas is `pointer-events: none` except while the cursor is over the pile (a window `pointermove` toggles it), so clicks elsewhere pass through. The stack layer registers itself with `BoardScene.setStackAnchorProvider` so stack-origin arrows, the casting arrow, and fly-from-stack seeds all resolve from live Pixi sprites — there are no stack DOM nodes to query. The stack is a pure overlay: it never reflows the board beneath it (no keep-out rect), and its right inset is fixed (the bottom-anchored action panel never shares horizontal space with it). Stack spec is built in `Game.tsx` (`stackSpec`) and threaded through `GameBoard`. The pile has a user **collapsed** state (`useStackUIStore.collapsed`): a left-side button collapses it (shown when expanded), and a pulsing right-edge button expands it (shown when collapsed). Collapsing slides the fan right with a fast decelerating ease until only the top card's border peeks at the edge (under the expand button) — it never fully vanishes; a new card landing while collapsed auto-peeks the fan out for ~1.2s; while collapsed, top-card arrows anchor to the expand button.
- **Layout** is a single arrangement: the local player along the bottom (`RegionOrientation` `bottom`), opponents across the top (`top`). Each opponent field's `rect` is the **fixed play area** — `computeBoardLayout` starts field `i` at `i · COLLAPSED_OPPONENT_WIDTH_PX` from the left and extends it to the canvas right edge, so its rect **equals its maximally-expanded clip band** (every field to its left collapsed to a banner). Since a delimiter can never push field `i`'s band-left below `i · COLLAPSED` (the grip clamp's `minGap`), the band is always a subset of the rect, so the felt/grid/cards align with the field's visible left edge when expanded and never leave a gap. The grid, card positions and felt are drawn from this fixed `rect` and **never move or reflow** (only the clip mask moves). The **playmat image is the exception**: `BoardRegion.bandZone()` fits it to the _visible_ band (usable zone ∩ clip), re-fitting on `setClip`, so it sits inside each field with equal padding on all sides instead of being anchored to the canvas-right-extending rect.
- **Delimiters live entirely in `BoardScene`, not React.** It owns `delimCurrent`/`delimTarget` (`count - 1` ascending fractions of width), eases them toward the target in its ticker (`DELIMITER_EASE`, tweakable at the top of `BoardScene`), and each frame applies the resulting **clip band** to each region via `BoardRegion.setClip(x, width)` — a mask-only call that never relayouts. The bands **TILE the canvas — they never overlap** — which is what stops one field's cards bleeding into another's (clipping alone can't; the felt is semi-transparent, so it's drawn over the fixed `rect` and only the mask moves). The **grip handles are Pixi** (`gripLayer`, draggable, `col-resize`); a drag sets one delimiter directly and overrides the ease until the next turn. React only sets the **target**: `BoardCanvas`'s `focusedOpponentId` prop → `scene.setOpponentFocus(playerId | null)` (an opponent's turn focuses their field; our turn → even split). `GameBoard` derives `focusedOpponentId` from `activePlayerId`; it holds **no** delimiter state.
- **Hover detection** is also `BoardScene`'s: it maps the cursor's canvas-x to the opponent whose clip band contains it, tints that battleground in the player's seat colour (`highlightGfx`, colour passed per-region via `BoardPlayerSpec.color`), and fires `onHoverOpponent(playerId | null)` for callers (`GameBoard` stashes it in `hoveredOpponentRef` for later use).
- **Pixi player HUD + on-grid zones**: the player panel is **all Pixi, always on** — there are no React player panels (the old `PlayerPanel`/`PlayerAvatar`/`BadgeOrbit`/`ZoneActionColumn` chain and the `pixiPlayerBar` pref were removed). It lives in `@/pixi/hud/`:
  - **`PlayerHudLayer`** owns one **`PlayerHudCapsule`** per player, a shared **`PlayerHudTooltip`**, and is positioned by `BoardScene` (`setRect`). Each capsule has a circular **avatar** (photo → bot game-icon → name initials) with the **life total in a pill straddling its bottom edge** (MTGA-style), and to its right a **two-row block**: hand-size badge + the floating **mana pool** (nonzero pips via `manaSymbolCache`) on top, all other **player/game badges** (glyphs via `gameIconTexture`, tinted per badge colour) below — the bottom row **wraps within `PLAYER_HUD_MAX_WIDTH_PX`** so it never bleeds into the hand. The panel itself is transparent (text carries a drop-shadow). The badge list, colours, and **labels** are built in `@/components/game/panels/playerHudBadges.ts` (`buildPlayerHudBadges`, mirroring the legacy React badge logic, plus self-only `useGameDevStore` overrides for preview) and passed on `PlayerHudSpec.badges`. Self capsule is bottom-left (`SELF_PLAYER_HUD_HEIGHT_PX`); opponent capsules sit across the top of their clip band, eased with the delimiters, and **reflow into a full-height vertical stack** (avatar + life + badges + mana down the column) when the field collapses to banner width — `BoardScene` gives the collapsed column the **whole band + the field's full height**. The solid occluding background is **not** drawn by the panel: `BoardScene.collapseVeil` paints a `appTheme.background` rect per opponent field whose **opacity ramps with the band width** (0 at ~2× the collapsed width → 1 at collapsed), computed in the same `applyDelimiters` pass that eases the delimiters — so it's frame-perfectly in sync with the collapse (no separate tween → no transition glitch). It sits above the cards, below the panels. Expanded fields get the fixed capsule height and the transparent capsule. **Commander damage** is shown per source: one crossed-swords badge per dealing commander, tinted with that opponent's seat colour (resolved in `GameBoard` via the commander card's `ownerId`), count in the lethal colour at ≥21. Tap a **targetable** avatar to target, otherwise to open its **`PlayerSheetModal`** (full per-player breakdown, reusing the spec's badges). The **self** capsule has a small **gear** on the avatar; tapping it fires `onShowBoardMenu` (plumbed like `onShowPlayerSheet`) which opens the React **`MiddleBarDock`** menu (controlled, anchored near the gear) with fullscreen / dev-panel toggle / concede. **Hover** a badge or the avatar for a `PlayerHudTooltip` (the Ring badge shows its ability list with the active level highlighted; clamps to the viewport). Per-player states: **eliminated** (in `gameView.concededPlayerIds`) → greyscale `ColorMatrixFilter` + skull + dim; **disconnected** (`RoomPlayerInfo.connected === false`) → dim + offline glyph; **inactive opponents** dimmed on others' turns. Avatar images load through `avatarTextureCache` (canvas → `ImageSource`, so data-URL uploads work where `Assets.load` can't). Animations are **gsap** (odometer life roll + floating delta, badge fade-in, priority pulse, targetable-ring pulse, turn-flash, life-loss damage wash, monarch/initiative sparkle burst) — no Pixi-ticker wiring. `TextStyle`s are shared via a `(size,weight,fill)` cache; capsules skip re-render when the spec signature is unchanged. A `sr-only aria-live` region in `GameBoard` mirrors life/turn for screen readers (Pixi has no DOM a11y). Driven by `BoardCanvas` `playerBars` / `showPlayerBars`; specs built in `GameBoard.playerBarSpecs`. **Experience / ticket counters** are on `PlayerDto` (`experienceCounters` / `ticketCounters`) but populated **only by the Java host** (`InteractiveSnapshotExtractor.toPlayer`); the Rust engine doesn't track them (`manabrew-engine/src/player/state.rs`) and serialises 0, so they show under Forge games and read 0 under the Rust engine until `PlayerState` gains the fields. Storm count is intentionally not surfaced (a per-turn game stat, not a persistent player counter).
  - **`BoardZoneTiles`** (owned by each `BoardRegion`) renders the **deck (card back) / graveyard / exile / command** zones as cards **on the grid**, occupying grid cells (a **vertical column at each player's back-left corner** by default via `BoardRegion.placeZoneTiles` — `nextDefaultCell` fills column-major over the mirror-aware `rowOrder`, so the stack grows away from the centre line and wraps into the next column past the field's row count; for mirrored opponents the column also extends into the reserved **attack-row band** (synthetic row `grid.rows`, positioned at `frontEdgeY`) so the stack flows into that inner-edge slot before wrapping; it reserves those cells so battlefield cards lay out around them). A non-empty zone shows the top card's `CardSprite` face (the MTG card back for the deck, loaded once via the shared `loadCardBack` memo so every region gets it) centred in the cell + a count pill; an **empty** zone is a dotted rounded-rect outline with the zone glyph (`ZONE_ICONS`). No labels. Tap-to-open + playable/target highlight for **all** players; the **local** player can drag a tile to another cell (drags paint the drop grid via `BoardRegion.drawDropGrid`, snap + persist in `BoardRegion.zoneSlots`), opponents are static. Driven by `BoardCanvas` `zoneTiles` (a `Record<playerId, ZoneTileSpec[]>`) → `BoardScene.setZoneTiles`.
- The player HUD no longer reserves space via DOM measurement: `BoardScene.setPlayerBlockers` is now fed **only** the local action-button cluster (`[data-action-cluster]`) as a keep-out. Player panels reserve their band height via the region top/bottom reserves instead.

The legacy per-player canvases (`PixiGameCanvas` / `PixiGameScene` / `PixiArrowsCanvas` / `PixiPhaseStripCanvas` / `OpponentHalf`) have been removed — do not reintroduce them.

## Modal pattern

All game modals use the `Modal` compound:

```tsx
<Modal onClose={onCancel} maxWidth="max-w-md">
  <Modal.Header>{/* title + optional source-card thumbnail */}</Modal.Header>
  <Modal.Instructions>Instruction text</Modal.Instructions>
  <Modal.Body>{/* content */}</Modal.Body>
  <Modal.Footer>{/* buttons */}</Modal.Footer>
</Modal>
```

Use `MODAL_CARD_THUMBNAIL` / `MODAL_CARD_IMAGE` / `MODAL_FOOTER_BETWEEN` constants. Use `useModalKeyboard` for Enter/Escape handling. Use `CardImageThumbnail` for header art.

For a modal that renders a list/grid of cards the user searches through (zone viewers, tutors, choose-from-zone), gate a name filter with `useCardNameFilter(cards)` (shows only past ~10 cards) and render `<ModalCardFilter>` between the header/instructions and `Modal.Body`, mapping its `filtered` instead of the raw `cards`. Pass `autoFocus` only when the modal has no Space/Enter-to-confirm (otherwise the focused input swallows the shortcut).

## Mana text

Any text that may contain `{W}`, `{2}{R}`, etc. renders through `TextWithMana`:

```tsx
<TextWithMana text={description} manaSize="sm" />
```

Don't roll your own symbol parsing.

## Sizing

Use the standard size constants. Don't invent pixel values.

| Constant             | Usage                                                                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `BATTLEFIELD_CARD`   | `w-[70px] h-[98px]` — battlefield (where React is involved)                                                                                 |
| `HAND_CARD`          | `w-[80px] h-[112px]` — hand / zone viewer                                                                                                   |
| `HAND_CARD_BASE`     | Hand-card base pixel dims, scaled at runtime by `useHandScale` (`HAND_FAN_PARAMS` in `pixi/HandLayout.ts` holds the fan spread/lift params) |
| `MODAL_CARD_SIZE`    | `w-[100px] h-[140px]` — cards inside modal grids                                                                                            |
| `MULLIGAN_CARD_SIZE` | `w-[160px] h-[222px]` — cards inside mulligan modals                                                                                        |
| `FLASH_CARD_SIZE`    | `{ w: 310, h: 434 }` (numeric — for Pixi-rendered preview, not a Tailwind class)                                                            |

## Card rendering — keep the three renderers in sync

A card's look is produced by **three independent renderers**. They share no draw code, so any visual change must be applied to all three (or consciously skipped with a note):

| Renderer                              | Tech | Where it shows                                                                                      |
| ------------------------------------- | ---- | --------------------------------------------------------------------------------------------------- |
| `@/pixi/CardSprite`                   | PIXI | In-game battlefield + hand (the real game)                                                          |
| `components/game/BattlefieldCardFace` | DOM  | Custom battlefield styles (art-forward / mini-frame) — settings preview, dev gallery (`/card-mock`) |
| `components/game/CardPreview`         | DOM  | Hover preview popup (and deck-editor hover)                                                         |

**Rule:** when you add, restyle, or remove a card visual — counters, damage wash, summoning-sickness, P/T coloring, frame tint/bars, keyword chips, mana pips, foil, badges, etc. — change **every** renderer that should show it, in the same commit. A change that lands in only one drifts the others (this has bitten us repeatedly: FPS-style regressions, counter display, damage-wash corners, summoning sickness).

- **Shared, theme-driven color math lives in `@/themes/gameTheme.ts`** (`frameTint`, `readableTextColor`, `withAlpha`, `darken`, `relativeLuminance`). Use these in both DOM and Pixi instead of re-deriving — that's what actually keeps the look identical across renderers. No hex literals (see Theme below).
- **Geometry must match too.** A full-card overlay must trace the card's rounded corner: in the DOM the card border is drawn as an `inset-0 rounded-[inherit]` overlay (not a CSS `border`) so washes can fill to the outer radius; in Pixi the equivalent is a `roundRect(..., CARD_RADIUS)`.
- **The realistic P/T badge is one shared component — `PtBadge`.** The bottom-right P/T overlay on the DOM renderers (`Card`, `CardPreview`) is `PtBadge`, sized in container units (`cqw`) + `em` so it is the same fraction of the card on every surface (stack, modals, zones, hover preview) regardless of pixel size. Its card root must set `@container`. Don't re-hard-code a px-sized P/T span — drift here is exactly what made the stack/preview/hand badges different sizes. **Hand & realistic-style cards don't draw it at all in Pixi** (`CardSprite.updatePT` is battlefield-only): the printed image already carries the P/T at the real proportion, so the overlay would double it mismatched. The art/mini-frame face (`BattlefieldCardFace`) lays P/T into its frame bar instead — an intentional, separate layout.
- **Attacking + summoning-sickness now match the DOM in Pixi too:** both draw an inset **edge glow** (`CardSprite.updateEdgeGlow`) — red for attacking, frosty + pulsing for summoning-sick (pulse driven per-frame by `BoardRegion.animate` → `tickEffects`); summoning-sick also keeps the grey desaturate/dim filter. Pixi's `setRing` is now reserved for **interaction/combat** states only (lethal, pending, tappable/untappable land, selectable/targeting).
- **Known intentional divergences — do not "fix" without being asked:** some effects are **Pixi-only** (doomed wash, foil ring, ring-bearer, status badges `MORPH`/`EXERTED`/…, damage-order badge, stack count). The DOM face also currently lacks the _debuffed_ P/T color that Pixi has.

## Prompt routing

The engine sends a `Prompt` (from `@/protocol`) → `PromptModalHost` (in `prompts/promptComponents.tsx`) looks up `prompt.input.type` in the `PROMPT_MODALS` registry and renders that entry inside `PromptModalController`. Each registry entry receives `{ prompt, respond, ctx }` — it reads the typed `prompt.input`, renders a leaf modal from `modals/`, and answers by calling `respond(<PromptOutput>)` directly (no per-prompt store callback). `ctx` carries the misc UI extras the engine prompt doesn't (`sourceDeckCard`); non-prompt overlays (zone viewer, spell stack, ability picker) live in `GameOverlays`, not the registry.

To support a new prompt: add `src/protocol/prompts/<name>.ts` (`Type`/`Input`/`Output`), add it to the unions in `protocol/prompts/index.ts`, add a `PROMPT_MODALS` entry, and build the modal under `modals/`. The Rust counterpart lives in `manabrew-rs/crates/manabrew-agent-interface/src/prompt.rs`.

## Theme

This subtree is the most color-heavy in the app. **No hex / rgba / palette tailwind.** Add a semantic key to `GameThemeColors` (in `src/themes/gameTheme.ts`), wire it in `buildGameColors()` (in `src/themes/buildGameColors.ts`), regenerate the Tailwind `@theme` block with `node scripts/generate-theme-css.mjs --write`, then use the generated token utility (`bg-pointer-hostile`, `text-counter-p1p1`, `ring-card-ring`, `bg-pt-buffed`, …). See `docs/agents/UI_THEME_RULES.md`.
