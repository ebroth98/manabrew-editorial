# GraalVM native Forge library

`build-native.sh` (macOS/Linux) and `build-native.ps1` (Windows) compile the
Forge harness into a native shared library (`libforgeharness.{dylib,so}` on Unix,
`forgeharness.dll` + import lib `forgeharness.lib` on Windows) via GraalVM
`native-image`, exposing the `ManaBrewEngineAdapter` over a small C ABI
(`ForgeNative.java`). `self-hosted-node` links it (feature `graal-forge`) to host
Forge rooms in-process ŌĆö no JVM, no jar subprocess.

## Prerequisites

1. **GraalVM for JDK 21** with `native-image` (Community Edition is fine).
   `native-image` ships in the distribution for JDK 21+ ŌĆö no `gu install` needed.

   Install any one of:
   - Download: the `jdk-21` build for your platform from
     <https://github.com/graalvm/graalvm-ce-builds/releases>, extract it, and
     point `GRAALVM_HOME` at `<dir>/Contents/Home` (macOS) or `<dir>` (Linux).
   - SDKMAN: `sdk install java 21.0.2-graalce`
   - Homebrew: `brew install --cask graalvm-jdk`

   `build-native.sh` defaults `GRAALVM_HOME` to
   `~/.local/graalvm/graalvm-community-openjdk-21.0.2+13.1/Contents/Home`;
   set the `GRAALVM_HOME` env var to override.

2. **JDK 17 + Maven** to build the harness fat jar (`yarn build:harness`).

## Build

macOS / Linux:

```bash
yarn build:harness          # JDK 17 + Maven ŌåÆ forge-harness/target/ŌĆ”jar
./forge-harness/build-native.sh   # GraalVM ŌåÆ native/build/libforgeharness.dylib
cargo build -p self-hosted-node --features graal-forge
```

Windows (PowerShell, from an MSVC dev shell ŌĆö "x64 Native Tools" prompt, so
`native-image` finds `cl.exe`; CI uses `ilammy/msvc-dev-cmd`):

```powershell
$env:GRAALVM_HOME = "C:\path\to\graalvm-jdk-21"   # dir containing bin\native-image.cmd
yarn build:harness                       # ŌåÆ forge-harness\target\ŌĆ”jar
.\forge-harness\build-native.ps1         # GraalVM ŌåÆ native\build\forgeharness.dll + .lib
cargo build -p self-hosted-node --features graal-forge
```

`self-hosted-node/build.rs` links `forgeharness` from `native/build/`; set
`FORGE_NATIVE_LIB_DIR` to link a copy from elsewhere. On Windows it also copies
`forgeharness.dll` next to the built exe (there is no rpath on Windows ŌĆö the
loader resolves the dll from the exe's directory). `native-image` cannot
cross-compile, so build the library on each target platform.

## What's tracked vs generated

Tracked (hand-authored): `forge/harness/ffi/ForgeNative.java`, `extra-config/`
(reachability metadata the agent and enumeration can't derive), `build-native.sh`,
`build-native.ps1` (the two must stay in sync).

Generated by `build-native.sh`, gitignored: `build/`, `classes/`, `gen-config/`
(class families enumerated from the jar), `native-image-config/` (tracing-agent
capture; cached ŌĆö delete it to recapture, e.g. after a Forge bump).
